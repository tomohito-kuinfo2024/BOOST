# StartOS 開発メモ

このファイルは、StartOS の最小試作品を作るために「何を用意したのか」「それぞれ何のためにあるのか」を後で見返すためのメモ。

## ここまででやったこと

このフォルダを React アプリとして動かせる形にした。

具体的には、以下を用意した。

- React アプリの基本ファイル
- Vite という開発用ツール
- このプロジェクト専用の Node.js / npm
- 開発サーバーを起動するスクリプト

まだ StartOS の本体機能はほとんど作っていない。
今は「React で画面を作り始められる土台」ができた状態。

## 各ファイル・フォルダの役割

### package.json

アプリの説明書。

アプリ名、使用ライブラリ、起動コマンド、ビルドコマンドなどが書かれている。

たとえばこのアプリでは、React や Vite を使うことが `package.json` に書かれている。

よく見る場所はここ。

```json
"scripts": {
  "dev": "vite",
  "build": "vite build",
  "preview": "vite preview"
}
```

意味は以下。

- `npm run dev`: 開発サーバーを起動する
- `npm run build`: 本番用にアプリをまとめる
- `npm run preview`: build したものを確認する

### package-lock.json

実際にインストールされたライブラリのバージョンを記録するファイル。

別のPCや別の日に同じ環境を再現しやすくするためにある。

基本的に自分で手書き編集しない。
`npm install` などを実行すると npm が自動で更新する。

### index.html

ブラウザが最初に読む HTML ファイル。

React アプリでは、HTML をたくさん手書きするのではなく、まずこの場所を用意する。

```html
<div id="root"></div>
```

この `root` の中に、React が画面を描画する。

また、以下の行で React の入口ファイルを読み込んでいる。

```html
<script type="module" src="/src/main.jsx"></script>
```

### src/main.jsx

React アプリの入口。

`App.jsx` に書いた画面を読み込んで、`index.html` の `root` に表示する。

大まかにはこういう役割。

```jsx
createRoot(document.getElementById('root')).render(
  <StrictMode>
    <App />
  </StrictMode>,
)
```

`<App />` が、実際に表示するメイン画面。

### src/App.jsx

画面の中身を書くメインのファイル。

今は仮の画面として、StartOS のタイトルと説明文だけを表示している。

今後、最初に触ることが多いのはこのファイル。

たとえば次のステップでは、ここに以下を追加していく。

- 今の気分を選ぶUI
- 使える時間を選ぶUI
- 候補を3つ表示するUI
- 各候補の「理由」と「最初の5分」

### src/index.css

アプリ全体に効く CSS。

文字の種類、背景色、全体の余白、ボタンや入力欄の基本設定などを書く場所。

全ページ共通の見た目を置くイメージ。

### src/App.css

`App.jsx` の見た目を整える CSS。

今の白いパネル、見出し、余白などはここで指定している。

画面単位のデザインは、まずここに書いていけばよい。

### node_modules/

`npm install` で入ったライブラリが保存されるフォルダ。

React、Vite、そのほか必要なパッケージがここに入っている。

とても大きくなりやすいので、Git では普通は管理しない。
そのため `.gitignore` に書いてある。

### .tools/

このプロジェクト専用の Node.js を置いているフォルダ。

最初、PC側にも WSL側にも `node` / `npm` が入っていなかった。
そこで、PC全体にインストールするのではなく、このプロジェクト内だけで使える Node.js を `.tools` に置いた。

これにより、StartOS の開発だけに必要な Node.js を使える。

このフォルダも Git では管理しない。

### scripts/dev.sh

開発サーバーを起動するためのスクリプト。

このプロジェクト専用の Node.js を使って、内部で `npm run dev` を実行している。

WSL のターミナルでは、基本的に以下を実行すればよい。

```bash
cd ~/projects/start-os
./scripts/dev.sh
```

### scripts/start-dev-background.sh

開発サーバーをバックグラウンドで起動するために用意したスクリプト。

ただし、今回の実行環境では WSL 内のバックグラウンドプロセスが終了しやすかった。
普段の開発では、まず `./scripts/dev.sh` を使えばよい。

### dist/

`npm run build` を実行したときに作られる、本番用ファイルの出力先。

これも自動生成されるので、Git では普通は管理しない。

## Vite とは

Vite は、React アプリを開発しやすくするための道具。

主に以下をやってくれる。

- 開発用サーバーを立てる
- React のコードをブラウザで動く形に変換する
- ファイルを変更したら画面を自動更新する
- 本番用にファイルをまとめる

開発中は以下のURLでアプリを見る。

```text
http://localhost:5173/
```

`localhost` は「自分のPCの中で動いているサーバー」という意味。

## Node.js と npm とは

React や Vite を使うには、Node.js と npm が必要。

### Node.js

JavaScript をブラウザの外、つまりPC上で動かすための実行環境。

React の開発ツールや Vite は Node.js 上で動く。

### npm

React や Vite などのライブラリを入れるための道具。

`npm install` を実行すると、`package.json` に書かれたライブラリをインストールしてくれる。

今回入ったバージョンは以下。

```text

npm 10.8.2
```

## 現在の上流設計メモ

StartOS は、動きたいのに動けない大学生が「最初の5分」を始めるためのアプリ。

現時点では、AI推薦・DB・ログイン・通知・カレンダー連携は入れない。
気分と時間の組み合わせから、固定の3候補を表示する。

### 気分データ

気分は `moodOptions` に配列で持たせている。

各気分は以下の形。

```js
{
  id: 'exhausted',
  label: '疲れ切っている',
  description: 'まずは休む前提で、開くだけでも十分な状態',
  motivationLevel: 1,
  aiSignals: {
    sourceText: '疲れ切っている。まずは休む前提で、開くだけでも十分な状態',
    activationNeed: 'lowest-friction',
    suggestionStyles: ['open-only', 'reduce-friction', 'no-pressure'],
  },
}
```

`motivationLevel` は 1, 2, 3 の3段階。

- `1`: かなり動きづらい。最初の5分のハードルを極限まで下げる。
- `2`: 動きたい気持ちはあるが、開始摩擦が大きい。軽い着手・整理・短時間タスクを優先する。
- `3`: 比較的動ける。理解・演習・創作など、少し負荷の高いタスクも候補に入れる。

画面には気分を3個ずつ表示する。
12個以上を一度に見せると読む負担が大きいため、最初の選択肢は少なくしている。
3個の内訳は、`motivationLevel` 1, 2, 3 から1つずつ。
各レベル内ではランダムに選ぶ。
該当する気分がない場合は「他の気分を表示」ボタンで次の3個に切り替える。
一度表示した気分は、同じ画面操作中には再表示しない。

将来的にAIでその人の傾向を見て気分候補を並べ替えられるように、各気分には `aiSignals` を持たせている。

- `sourceText`: AIが意味を読み取りやすいように、ラベルと説明文をまとめた文章
- `activationNeed`: どれくらい低いハードルが必要か
- `suggestionStyles`: 提案の方向性

### 時間データ

使える時間は `timeOptions` に配列で持たせている。

現在の選択肢は以下。

- `5分`
- `15分`
- `30分〜`

### 候補データ

候補は `suggestionTemplates` に `motivationLevel` ごとに3つずつ持たせている。

現在は以下の流れ。

1. ユーザーが気分を選ぶ
2. 選んだ気分から `motivationLevel` を取得する
3. ユーザーが使える時間を選ぶ
4. `createSuggestions()` で固定の3候補を作る
5. 各候補に「理由」と「最初の5分」を表示する

将来的に今日のご褒美や直近のワクワクイベントを反映できるように、`createSuggestions()` は `personalContext` を受け取る形にしている。

```js
const futurePersonalContext = {
  excitingEvent: '',
  recentMoodIds: [],
  frequentMoodIds: [],
  preferredSuggestionIds: [],
}
```

`reward` は画面の「今日のご褒美」入力欄から受け取る。
空欄の場合は提案カードに表示しない。
入力がある場合だけ、提案カード内に「少し進めると、ご褒美をより気持ちよく楽しめる」という文脈で表示する。

ご褒美はタスク完了の義務条件ではない。
ユーザーを責める文言や、「終わらせないと楽しめない」という文言は使わない。

「何か提案して！」ボタンを押すと、`rewardSuggestions` からランダムに1つ入力欄へ入る。
押すたびにできるだけ別の候補を入れる。
フィルイン後も、ユーザーは自由にテキストを書き換えられる。
候補は「ラーメン行く」「服を見に行く」など、敷居の低い内容を中心にしている。

今はまだAI連携はない。
後で `excitingEvent` に入力値を入れ、提案文に差し込めるようにする予定。
さらに、後で `recentMoodIds` や `frequentMoodIds` を使って、よく選ぶ気分に近い候補を優先表示できるようにする予定。

## 課題登録・進捗管理の追加方針

次の段階では、具体的な課題や目標を登録し、それをもとに提案を出す。

一気に全部作らず、以下の順で進める。

1. 型定義と固定選択肢を作る
2. localStorage で課題を保存・読み込みできるようにする
3. 課題登録画面を作る
4. 課題一覧画面を作る
5. メイン画面の提案カードで登録済み未完了課題を使う
6. 「これをやる」から作業中状態を作る
7. 振り返りログを localStorage に保存する

### src/domain.js

`src/domain.js` は、StartOS 内で使うデータの形と固定選択肢をまとめるファイル。

このプロジェクトはまだ JavaScript で作っているため、TypeScript の `type` ではなく JSDoc コメントで型のメモを書いている。

ここには以下を置いている。

- `DangerLevel`: `S`, `A`, `B`, `C`
- `TaskGenre`: 試験勉強、レポート、課題など
- `TaskProject`: 登録する課題・目標の形
- `ReviewLog`: 作業後の振り返りログの形
- `dangerLevelDescriptions`: 危険度ごとの説明文
- `startEaseOptions`: 開始しやすさの5段階
- `taskGenres`: 登録画面で使うジャンル一覧
- `reviewRatingOptions`: 1〜5 の自己評価選択肢

このファイルを先に作っておくと、登録画面・一覧画面・レビュー画面で同じ選択肢を使い回せる。

### src/storage.js

`src/storage.js` は、ブラウザの `localStorage` にデータを保存・読み込みするためのファイル。

`localStorage` は、ブラウザ内に小さなデータを保存できる仕組み。
DBやログインを使わなくても、同じブラウザで開く限りデータを残せる。

今は以下の2種類を保存できるようにしている。

- `TaskProject[]`: 登録した課題・目標の一覧
- `ReviewLog[]`: 作業後の振り返りログ一覧

主な関数は以下。

```js
getTaskProjects()
saveTaskProjects(taskProjects)
addTaskProject(taskProject)
updateTaskProject(taskProjectId, updateTaskProject)
deleteTaskProject(taskProjectId)

getReviewLogs()
saveReviewLogs(reviewLogs)
addReviewLog(reviewLog)
```

保存するときは、配列を JSON 文字列に変換して `localStorage` に入れている。
読み込むときは JSON を配列に戻す。

もし保存データが壊れていた場合でも、アプリが落ちないように空配列を返すようにしている。

### 課題登録画面

上部タブは「行動開始」と「課題一覧」の2つ。
課題登録画面へは、課題一覧画面の最上段にある「課題を登録」ボタンから移動する。

課題登録画面では以下を入力できる。

- タイトル
- 期限
- 危険度
- 開始しやすさ
- ジャンル
- やるべきタスク
- メモ

タイトルは必須。
期限・やるべきタスク・メモは任意。

期限は以下から選べる。

- `カレンダー`: 日付を指定する
- `長期目標`: 明確な締切より、長期的に進めたい目標として扱う
- `期限なし`: 期限を設定しない

保存すると `TaskProject` の形にして `localStorage` に保存する。

```js
{
  id: 'task-...',
  title: '信号とシステム期末勉強',
  deadline: '2026-07-20',
  deadlineType: 'date',
  dangerLevel: 'S',
  startEase: 2,
  genre: '試験勉強',
  subTasks: ['スライド確認', '演習問題'],
  memo: 'ラプラス変換が不安',
  isCompleted: false,
  completedAt: undefined,
  createdAt: '...',
  updatedAt: '...',
}
```

危険度の説明文とジャンル一覧は `src/domain.js` から読み込んでいる。
保存処理は `src/storage.js` の `addTaskProject()` を使っている。

危険度は `select` ではなく、S/A/B/C の説明文つきボタンで選ぶ。
選ぶ前に説明を読めるようにするため。

危険度は「やるべき圧」を表す。
開始しやすさ `startEase` は「今始められるか」を表す別軸。

開始しやすさは 1〜5 で選ぶ。

- `1`: かなり重い
- `2`: やや重い
- `3`: 普通
- `4`: 比較的始めやすい
- `5`: かなり始めやすい

### 課題一覧画面

登録した課題は「課題一覧」タブで確認できる。
課題一覧画面の最上段には「課題を登録」ボタンがあり、そこから登録画面へ移動できる。
課題一覧画面で下にスクロールすると「ページトップへ」ボタンが表示され、最上段へ戻れる。

表示する内容は以下。

- タイトル
- 完了状態
- 危険度
- 開始しやすさ
- ジャンル
- 期限
- メモ
- やるべきタスクの一部

操作は以下。

- `編集`: 課題登録フォームを再利用して内容を修正する
- `完了にする`: `isCompleted` を `true` にする
- `未完了に戻す`: 完了済みの課題を戻す
- `削除`: 確認ダイアログを表示してから、localStorage から課題を削除する

課題一覧では、未完了タスクを上部、完了タスクを下部に分けて表示する。
登録直後は課題一覧タブに移動し、今追加した課題カードが画面中央あたりに映るようにスクロールする。

未完了タスクは、期限が近いものを上に表示し、期限が同じ場合は危険度が高いものを上に表示する。
期限なしの課題は、期限ありの課題より下に表示する。

完了タスクは、完了した日時が新しいものを上に表示する。
完了時には `completedAt` を保存する。

メイン画面のおすすめフローでは、未完了の課題だけを使う。

以前は候補を3つ並列表示していたが、選択肢そのものが負担になる可能性があるため、今は順序付きのフローとして表示する。

```text
Step 1: まずこれ
Step 2: 次にこれ
Step 3: 余力があればこれ
```

フローは固定ルールで作る。

- 気分が低い場合は、Step 1 に `startEase` が高い課題を置く
- 気分が低い場合の Step 1 は、動き出すためのブースターとして扱う
- 気分が低い場合でも、Step 2 には危険度が高い課題や期限が近い課題を置く
- 気分が比較的高い場合は、Step 1 から危険度が高い課題や期限が近い課題を置く
- Step 3 は余力があれば触れる課題として置く
- Step 3 では、期限が2か月以上先のタスク、`長期目標`、`期限なし` を同じ優先枠として扱う

各Stepには以下を表示する。

- 課題タイトル
- 期限
- ジャンル
- やるべきタスク
- メモ
- 理由
- 最初の5分
- 今日のご褒美が入力されている場合の文言
- 進捗を更新ボタン

気分の `motivationLevel` と使える時間によって、おすすめフローの並び順と「最初の5分」の重さを変える。

使える時間は以下の3つ。

- `30分`: 始めやすさを重視し、小さな一区切りまで進める
- `60分`: 危険度・期限も見ながら、優先度の高い課題に向き合う
- `90分〜`: 重めの課題も含めて、まとまった進捗を作る

おすすめフローの v1 ルールは以下。

- 30分の場合は、始めやすさとバランスを重視する
- 60分の場合は、危険度・期限と始めやすさのバランスを見る
- 90分〜の場合は、危険度・期限をより重視し、少し重い課題も候補に入れる
- ただし `motivationLevel = 1` のときは、時間が長くても Step 1 に始めやすい課題を置く

「最初の5分」は、課題のジャンルごとに文章を変える。
テンプレートは `src/App.jsx` の `firstFiveMinuteTemplatesByGenre` にまとめている。

各ジャンルには以下の4種類の文を持たせている。

- `low`: 気分がかなり低いとき
- `short`: 30分の作業枠
- `medium`: 60分の作業枠
- `long`: 90分以上の作業枠

実際にどの文を使うかは `getFirstFiveMinuteTemplateType()` で決める。
その後、`createFirstFiveMinutes()` が `{target}` に最初のやるべきタスクを差し込んで表示文を作る。

`motivationLevel = 1` のときだけ、課題に入る前に `Step 0: リフレッシュ` を表示する。
これは、そもそも作業できる状態ではない可能性があるため。

`Step 0` の文章は `src/App.jsx` の `refreshStepTexts` にまとめている。
気分の `id` ごとに以下を書き換えられる。

- `title`
- `description`

`Step 0` は課題カードではないので、「理由」「最初の5分」のタグは表示しない。
タイトルと説明文だけを表示する。

おすすめフローから課題を編集した場合は、保存後に行動開始画面へ戻る。
課題一覧から課題を編集した場合は、保存後に課題一覧画面へ戻る。

おすすめフロー上では直接完了にしない。
「進捗を更新」から課題更新画面に入り、更新ボタンの横にある「完了にする」または「未完了に戻す」を選ぶ。

### お楽しみカレンダー

少し先の楽しみな予定を登録できるタブを追加した。

入力する情報は以下だけ。

- 日付
- タイトル

保存すると `FunEvent` の形で `localStorage` に保存する。

```js
{
  id: 'fun-event-...',
  date: '2026-07-10',
  title: '友達とラーメン',
  createdAt: '...',
  updatedAt: '...',
}
```

保存・読み込み・削除は `src/storage.js` の以下の関数で行う。

- `getFunEvents()`
- `addFunEvent()`
- `deleteFunEvent()`

行動開始画面では、おすすめフローの直前に直近のお楽しみ予定を1件表示する。
直近予定は、今日以降の日付の中で一番近い予定を選んでいる。

この予定はタスク完了の義務ではなく、「少し先に楽しみがある」と思い出すための情報として扱う。
